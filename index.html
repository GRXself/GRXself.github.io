在TWI的日子

5月 26, 2019 发布在 随感

写在前面的话：其实这篇原本是作为日记，写给自己看的（平常的日记也大都是在某个阶段有了什么感悟，随便写写，给以后的自己看），但是作为在TWI最开始的一段经历，我觉得分享出来也不错。所以怎么说呢，这篇就作为随感扔出来啦，希望自己的经历能对读到这篇的人也有所帮助，写的很烂，请见谅

到今天（20190526）为止，三天的TWI就结束了。以前总是觉得充实的日子很快就会过去，但是这三天却感觉不止过了三天，我更觉得是接受了一段洗礼。

第一天

第一天算是最累的一天，早上坐车去办公室，在车上我没有老老实实靠在椅背上，而是想跟更多的人说说话，结果我的颈椎果然没让我失望，到了之后就有点小头疼。我们在5楼群面的地方开始此日的TWI。两位讲师，袁哥和锋哥，带着我们先了解了TW的方方面面，价值观，文化，公司架构，历史。这让我明确了一点，我的选择没有错。

然后在中午，本来是预定在晚上7点到9点的闪电演讲，但是计划临时发生了变更，原定在今晚七点到九点开始的闪电演讲被分成了两部分，有一部分人需要中午就开始演讲。（共21人，最后的安排是今天中午10人，晚上数人，第二天数人）我当时抱着自己准备万全（第一次准备了逐字稿，还加了动作之类的备注）的心态，想着中午就开始我的表演（演讲），所以就在白板上把自己的名字写在了中午的区域。结果到了中午，我开始准备的时候，发现昨晚还算流畅的练习却怎么也重现不了了。我的焦虑不断上升，对内容的记忆也越来越混乱，到最后整个脑子仿佛被放到了夏天的柏油马路上。结果可想而知，我在演讲的中途卡壳了。我虽然带着拍了完整稿子的手机，但是模糊的画面加上紧张的心情，我根本找不到我断在哪里。我当时内心只剩下一个想法，这绝对是我人生中最差的一场演讲了。我的准备不够充分，内容过于臃肿（演讲时间5分钟，我没有选择使用PPT把控节奏），状态也不好（晕车，中午犯困），自己临场忘词的处理也做的出奇的差，虽然最后袁哥多给我了30s让我收尾，但是我还是做的太差了。我想说我很坚强，但是不受打击是不可能的。在我演讲之前，我还乐意去评判别人演讲的不足，但是在我自己做了糟糕的演讲之后，我自己总觉得无论我去说什么，都会显得无力（虽然大家可能不会那么在意，但是当时我自己已经蔫了）。我尝试去调整自己的心态，但是真的很难。

在中午的演讲结束之后，好像是锋哥做了总结，告诉了我们这次演讲的目的，是为了让我们提升自己的演讲能力，所以这次的表现并不重要，重要的是能从中学到什么。虽然在听了总结之后我并不能完全释怀自己的差劲表现，但是心态还是有了很大的回升。

下午，另一位讲师做了老人讲故事的环节。（非常抱歉，忘记了讲师的名字，只记得第二个字很复杂）讲师讲了从进入TW到今天这六年的故事，也让我看到了未来在TW能做的很多事，我个人觉得是非常有意义的一个环节。但是这个环节也有着一些缺点。首先是持续时间太长，一小时四十分钟没有休息，而且关了灯之后暗下来的环境让人很容易犯困，最后座位。本来大家是面向讲台的，但是袁哥想让大家能和讲师靠的更近，所以让我们围成了一个圈。但是场地限制导致我们围成了一个苹果的形状，讲师在苹果把的位置，我算幸运，坐在了面朝讲师的位置，所以能全程跟随讲师，但是有近三分之一的同学只能看到讲师的背影或者侧面，在很侧面的位置效果也不是很好，所以整个环节下来，大家都表现的很困。（感谢我自己坐在了第一排，所以幸运的能跟随讲师听完了整场演讲）这之后我们做大巴返回了酒店。

晚上，我们进行了聚餐，但是由于时间关系并不足以完成剩下的所有闪电演讲，所以晚上的闪电演讲只进行了很少的几个人。之后我和我的酒店舍友韩朝峰就回了房间，因为第一天真的很累（对我来说尤其是坐车）所以我俩就洗洗睡了。

第二天

第二天主要是做游戏，目的是让我们了解敏捷和精益。上午我们54号文件（扑克），叠帽子，画画，下午玩乐高（最后让破产的CEO袁哥上了天台233），学到了很多东西，也很开心。上午开始之前，把剩下的闪电演讲也做完了，锋哥也对演讲做了总结和指导。而下午开始前，我们做了灵魂伴侣的游戏，大家也有了更多的相互了解。

第二天的晚上，西安的十个小伙伴一起在马庆的房间玩了狼人杀。我以前其实基本没当过法官，这次也有好几个新人，所以我就自告奋勇当起了前几轮的法官。大家都玩得很开心，本来不会的人甚至在结束的时候还想再来几把。（我当然又当了一次MVP狼，哈哈）。

第三天

然后就是到了第三天，也就是真正的今天。传说中的考试来临了，不过也并没有完全出乎意料（毕竟已经经历了两天的洗礼了，2333）我们还是跟昨天玩游戏一样分了4组，然后进行了紧张激烈的19个题的抢题和答题。在我们都以为全部19题答完，今天的考核结束的时候，我们发现自己错了。压轴大戏演讲来了，3+3min（准备+演讲），现场给定主题，小组选一位进行演讲。我们组是最后一个，在开始的时候我其实不是很愿意去做这次演讲，因为才刚刚经历了闪电演讲的大失败，所以内心是希望有其他人能做代表的。但是大家并没有因为我之前的大失败而对我失去信心，真的很感谢大家的新任。我们组是最后一个演讲的，我们抽到的题目是假如时光可以倒流。我其实是一个不会后悔的人（可能还是会后悔啦，但是并不觉得需要回到过去去改变），我选择了我写日记开始的原因，我在大学本科的那段不算完美的记忆。我讲了自己鼓起勇气去表白，但是因为害羞，在表白之后却连邀请自己喜欢的人一起做什么事情都没有（其实还是有参加舞会啦，当时忘记说了，但也只有这一个主动邀请）最后被告知了我早已知道的结果，no。但是从那之后，我确实成长了很多，她启发了我，我开始选择去做自己。所以如果如果时光倒流我会做什么呢？我还是会跟以前一样去表白，只是这次我多了勇气。但是我知道现实中并没有再一次，我也不后悔自己当初的表现，因为今天我站在这里，都是因为我曾经的无力，曾经的胆小和懦弱。人生没有如果，我也不需要如果，我会努力做好现在的自己，未来还有更多需要我去体验的事情，希望以后看到这篇日记的我也能珍惜当下自己所拥有的一切，不要去后悔任何已经发生了的事。

光驱异常导致的数据读取异常

5月 14, 2019 发布在 硬件

在遇到我之前写的这篇文章(使用zip4j遇到的CRC错误（invalid CRC for file）———“_”字符造成的异常)的问题之后，又遇到了其他字符出错的问题，这次我发现了这个项目中问题的根源。

此项目中接收并读取压缩包的电脑光驱存在问题，导致其在读取某些光盘是会出现乱码（文件系统的名字已经出现异常，本来是普通的中文名称，变成了类似于__DS332S0这样的名称，不管是文件夹还是里面的文件都是如此）此乱码造成的影响不只局限于名称，读取的文件也受到了影响。在正常的光驱中读取的数据无论是名称还是内容均正常，而此问题光驱最终导致了异常的发生。

最终的解决方案就是不再使用此光驱读取光盘中的数据，借用其他光驱读取后利用其他移动存储介质转移到此电脑后再使用。

Java Overflow and Underflow

5月 14, 2019 发布在 java

今天在leetcode上做练习的时候遇到了一个回文数的题，考虑到不将整数转换为字符串来解决这个问题，我用求余把原整数倒序，然后跟原整数进行比较。

虽然我的做法通过了测试用例，也提交成功了，但是在我看官方题解时发现题解中考虑了直接倒序可能会出现临界情况，倒序的数可能会超过int的范围。官方题解中是只倒序了一半数字，然后对一半进行比较，所以没有涉及遇到边界的计算，但是我自己的solution涉及到了边界。

这里最大的问题是，我虽然没有考虑边界，但是却通过了测试。我开始以为是用例中没有包含边界，于是自己对边界进行了测试，但是实际测试之后才发现遇到边界之后java并没有抛出任何异常。

我对边界情况进行了调试，发现当一个边界内的数在数学运算之后超过边界后，其结果是可以预测的，但是却没有任何异常，也不是真正的数学运算结果。我在google和so上查找一番之后发现这种情况是数学运算之后，只有后32位的二进制数得到了保留。

在进一步的搜索之后，我找到了这篇文章,里面提到了Java Overflow And Underflow，而我遇见的情况就是overflow。

Java本身（我使用的是Java8）对这种overflow并没有任何runtime exception，所以这个问题只能靠自己去发现，需要保持警惕。

使用zip4j遇到的CRC错误（invalid CRC for file）———"_"字符造成的异常

11月 24, 2018 发布在 解压工具

今天遇到了一个棘手的解压问题，先描述一下大致的业务情况。压缩包是自己写的C#程序生成的（有加密），使用的是ICSharpCode.SharpZipLib，然后C#程序也可以解压这个加密的压缩包。这个加密的压缩包最终会上传到javaweb服务器上（springboot+tomcat+MySQL），javaweb项目使用的是zip4j。

结果客户提交上来的用C#程序生成的压缩包，无论是在C#程序上还是在java程序上都不能顺利解压（总共有200多文件夹，包含文件，但是解压到二十多个文件夹的时候出了问题）。

我尝试使用7Zip这个软件直接解压压缩包，发现压缩包可以正常解压，也不会报错。

在这之后我又使用我的C#程序对7Zip解压出来的文件重新进行压缩，然后发现我重新生成的文件在C#和java两边都能够正常解压。

难道是客户提交的压缩包有问题？但是7Zip解压的时候没有任何报错啊！

于是我开始在java程序那边打了断点进行调试，异常是：net.lingala.zip4j.exception.ZipException: net.lingala.zip4j.exception.ZipException: invalid CRC for file: XXXX(此处省略文件名）

然后我在zip4j找到了两个讨论CRC问题的贴子#贴子1、 #贴子2，发现zip4j最新版本（1.3.2）确实存在CRC的bug。

但是奇怪的是C#版本为什么也会有类似的问题？于是我用7Zip的file manager查看了用户提交的压缩包（存在异常）和我自己重新生成的压缩包（一切正常），在仔细对比之后发现异常压缩包停止解压的文件名包含_，用file manager查看发现异常的压缩包文件名中的_显示为?，而正常的压缩包则正常显示_。

由于客户那边周末休假，所以无法得知客户对压缩包做过什么，也暂时无法获取用户的系统情况，但是我猜测可能是由于客户系统编码的原因导致_字符出现了异常，所以目前的解决方案是禁止客户在命名的时候使用_。

此问题根源与zip4j无关，如需了解造成文中问题的根源请移步另一篇文章光驱异常导致的数据读取异常

初次使用dubbo框架遇见的坑

6月 13, 2018 发布在 分布式系统

按照dubbo首页的quick start入门，provider可以正常启动，但是consumer一直报错。
按照Demo写的代码运行Cunsummer时 Failed to check the status of the service com.alibaba.dubbo.demo.DemoService,找到了一个貌似指出问题的blog，里面提到

根本原因是源码中会判断发送给consumer的服务消息是采用单播还是组播，因为虚拟网卡对ip地址的影响导致判断逻辑出错，误认为consumer可以接收到单播，但其实consumer在收到单播消息之前provider自己就把这个消息给抢着处理了（因为provider和consumer在同一台机器上，相当于provider自发自收），导致consumer无法获取服务消息。此问题在linux系统上实验时又无法复现，因为linux和windows的接口对待是否回环是不同的，linux下的consumer可以在provider之先获取到单播消息。

并给出了解决方法：

1、禁用虚拟网卡。
2、在两台机器上分别部署
3、采用其它registry
4、将服务提供者的spring配置文件中改为(注意url后面的参数配置)
<dubbo:registry address="multicast://224.5.6.7:1234?unicast=false"/>

但是1、2、4在分别试过后都没用，最后我选择使用zookeeper来解决，也就是3中的方法，最后总算成功。但是dubbo首页给的方法真的很坑，muticast居然会报错，虽然根本原因可能是由windows系统引起的，以后需要仔细了解下多播，看看到底是怎么回事。
